# Token + Latency Optimization Policy (Jarvis)

This document captures the current policies and implementation notes for reducing token spend and improving perceived latency in the `idc1-assistance` Jarvis deployment.

## Goals

- Reduce tokens sent to and generated by the model per user interaction.
- Reduce end-to-end latency (especially on reconnect and tool-heavy flows).
- Preserve correctness and cross-device consistency (e.g. reminders via Weaviate).

## What usually drives token cost

- Tool declaration payload size (function schemas) on each Live session connection.
- Verbose tool outputs returned to the model.
- Re-sending large instructions / repeated context.
- Memory injection that is too broad (e.g. dumping large histories instead of targeted slices).
- Repeated embedding calls for identical or near-identical content.

## Current implemented optimizations

### 1) Tool schema minimization (connect-time token reduction)

On each `WS /ws/live` connect, Jarvis sends `function_declarations` to Gemini Live. Large tool schemas increase tokens and reconnect latency.

Policy:
- Expose the smallest set of tools required for the current deployment.
- Prefer enabling tools by allowlist.

Implementation:
- Env var: `JARVIS_TOOL_ALLOWLIST`
  - Comma-separated list of tool names.
  - When set, only allowlisted tools are declared to the model.

Suggested production allowlist (reminders-focused):
- `reminders_list`
- `reminders_upcoming`
- `reminders_done`
- `pending_list`
- `pending_confirm`
- `pending_cancel`

### 2) Embedding cache (latency + token reduction)

Weaviate upserts embed `kind + title + body`. Repeated upserts of identical content cause repeated embedding calls.

Policy:
- Cache embeddings for identical input text.
- Keep cache bounded.

Implementation:
- Env var: `JARVIS_EMBED_CACHE_MAX` (default `512`)
  - Maximum cached entries.
- Backend uses a SHA-256 keyed in-memory cache.

Notes:
- This cache is per-backend-instance (not shared across replicas).
- It improves resync/retry flows and repeated reminder updates.

## Recommended (next) optimizations

### A) Reduce tool response verbosity

Policy:
- Tools should return minimal fields unless more is explicitly needed.
- Prefer IDs + essential metadata.

Examples:
- For reminder completion, return `{ok, reminder_id, changed}` instead of full reminder bodies.

### B) Context windowing / rolling summaries

Policy:
- Avoid unbounded growth of conversation context.
- Maintain a rolling summary of older turns.

Implementation direction:
- Store a session summary keyed by `session_id`.
- Periodically update the summary (e.g. every N turns or when context exceeds a size threshold).

### C) Targeted memory injection

Policy:
- Prefer retrieving only the relevant window (e.g. next 48h reminders + recent fired reminders) rather than full history.

## Provider-level caching ("token cache")

Some providers support server-side prompt caching for repeated static prefixes. If/when Gemini exposes a stable caching mechanism for Live sessions, adopt it for:

- Tool declaration schemas (if supported).
- Static instruction blocks.

Until then, the most reliable wins are minimizing what we send and avoiding repeated work.

## Verification checklist

- Reconnect latency:
  - Measure time from WS connect to `{type: "state", state: "connected"}`.
  - Compare before/after applying `JARVIS_TOOL_ALLOWLIST`.

- Embedding load:
  - Observe reduced embedding call frequency during reminder resync/upserts.
  - Confirm Weaviate writes still occur and schema remains correct.

- Functional correctness:
  - Reminders:
    - `GET /reminders/upcoming` returns `pending` reminders.
    - `POST /reminders/{reminder_id}/done` marks reminder done and it disappears from upcoming.

## Configuration quick reference

- `JARVIS_TOOL_ALLOWLIST`: shrink declared tools to reduce connect-time tokens.
- `JARVIS_EMBED_CACHE_MAX`: bound the embedding cache.
